# Production-Grade Connector Framework Design

If the workflow engine is the heart of the platform, the connector framework is the operating system. A poorly designed connector framework becomes impossible to maintain once you reach 50–100 integrations.

The design below is inspired by patterns used by workflow platforms, integration platforms, API gateways, and plugin ecosystems, while being optimized for a TypeScript + PostgreSQL + Redis + BullMQ stack.

---

# Design Goals

The framework must support:

✅ Marketplace connectors

✅ First-party connectors

✅ OAuth2

✅ API Keys

✅ JWT

✅ Webhooks

✅ Polling triggers

✅ Actions

✅ Multi-tenancy

✅ Versioning

✅ Sandbox execution

✅ Connector upgrades

✅ Dynamic UI generation

✅ Backward compatibility

# Core Concepts

## Connector

A connector is a deployable plugin.

```typescript
interface ConnectorPackage {
  manifest: ConnectorManifest;

  auth: AuthProvider;

  actions: ActionDefinition[];

  triggers: TriggerDefinition[];

  webhooks?: WebhookDefinition[];
}
```

---

# Connector Manifest

The manifest drives:

* Marketplace
* UI rendering
* Versioning
* Compatibility

```typescript
interface ConnectorManifest {
  id: string;

  name: string;

  displayName: string;

  version: string;

  author: string;

  category:
    | "crm"
    | "database"
    | "payments"
    | "communication";

  sdkVersion: string;

  icon: string;

  description: string;

  permissions: string[];

  capabilities: CapabilitySet;
}
```

# Action System

```text
Create Contact
Update Lead
Send Message

```

---

## Action Definition

```typescript
interface ActionDefinition {
  id: string;

  name: string;

  description: string;

  inputSchema: JSONSchema;

  outputSchema: JSONSchema;

  execute(
    ctx: ExecutionContext,
    input: unknown
  ): Promise<unknown>;
}
```

---

## Example

```typescript
defineAction({
  id: "create_contact",

  name: "Create Contact",

  inputSchema: {
    type: "object",
    properties: {
      email: {
        type: "string"
      }
    }
  },

  async execute(ctx, input) {
    return hubspot.contacts.create(
      input
    );
  }
});
```

# Incremental Sync Support

Every polling trigger stores a cursor.

```typescript
interface PollingState {
  lastCursor: string;

  lastRunAt: Date;
}
```

Example:

```json
{
  "lastCursor": "ticket_12345"
}
```

Avoids duplicate processing.

# Credential Abstraction

Connectors never access secrets directly.

```typescript
const token =
 await ctx.credentials.get();
```

Under the hood:

```text
Vault
  ↓
Credential Service
  ↓
Temporary Token
  ↓
Connector Runtime
```

# Execution Context

Every connector execution receives context.

```typescript
interface ExecutionContext {
  tenantId: string;

  executionId: string;

  workflowId: string;

  logger: Logger;

  credentials: CredentialProvider;

  http: HttpClient;
}
```

---

# Built-In HTTP Client

Never allow raw fetch.

Provide:

```typescript
ctx.http.get()

ctx.http.post()
```

Benefits:

* Rate limiting
* Retry logic
* Tracing
* Metrics
* Secret masking

---

# Sandbox Architecture

```text
      ↓
```


# Version Compatibility

Never mutate old versions.

```text
HubSpot 1.0.0
HubSpot 1.1.0
HubSpot 2.0.0
```

Workflow references:

```typescript
{
  connectorId: "hubspot",
  version: "1.1.0"
}
```

Execution always uses the referenced version.

# Dynamic Form Generation

The builder UI should be generated from schemas.

```typescript
inputSchema: {
  type: "object",

  properties: {
    email: {
      type: "string"
    },

    company: {
      type: "string"
    }
  }
}
```

Automatically becomes:

```text
Email Input
Company Input
```

No custom frontend work required.
