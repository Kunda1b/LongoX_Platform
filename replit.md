# LongoX

A low-code automation + internal tools platform for SMBs — workflow automation (n8n-style) with a connector marketplace, plus a drag-drop internal tools builder.

## Run & Operate

- `pnpm --filter @longox/api-server run dev` — run the API server (port 8080)
- `pnpm --filter @longox/flowcraft run dev` — run the frontend (port 25075)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm run codegen` — regenerate all code: Orval hooks/schemas + gateway OpenAPI types + GraphQL resolver/operation types
- `pnpm --filter @longox/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @longox/api-gateway run codegen` — regenerate GraphQL resolver types + gateway OpenAPI TypeScript types
- `pnpm --filter @longox/shared-types run codegen` — regenerate `gateway-api.d.ts` from the gateway OpenAPI spec
- `pnpm --filter @longox/db run push` — push DB schema changes (dev only)
- `pnpm --filter @longox/shared-db run prisma:validate` — validate Prisma schema
- Required env: `DATABASE_URL` — Postgres connection string

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- Frontend: React + Vite + Tailwind CSS + shadcn/ui, served at `/`
- API: Express 5, served at `/api`
- DB: PostgreSQL + Drizzle ORM (compatibility layer) + Prisma (canonical, long-term)
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec) + openapi-typescript (gateway boundary types)
- GraphQL: graphql-yoga + @graphql-codegen (resolver types, operation types)
- Build: esbuild (CJS bundle)

## Where things live

### Architecture Contracts (Section 29.2)
- `services/api-gateway/openapi/openapi.yaml` — **canonical** OpenAPI 3.1 spec for the gateway boundary (Section 13); never hand-write types, run codegen
- `services/api-gateway/openapi/generated/gateway-api.d.ts` — gateway types generated by `openapi-typescript` (in api-gateway package)
- `packages/shared-types/src/generated/gateway-api.d.ts` — same types re-exported through shared-types for cross-package use
- `packages/shared-db/prisma/schema.prisma` — **canonical** Prisma schema for all 40+ tables (Section 10); source of truth for new development
- `services/api-gateway/graphql/schema.graphql` — GraphQL SDL (Section 14)
- `services/api-gateway/src/graphql/generated/types.ts` — base GraphQL TypeScript types (auto-generated)
- `services/api-gateway/src/graphql/generated/resolvers.ts` — typed resolver map with `GraphQLContext` (auto-generated)
- `services/api-gateway/src/graphql/generated/operations.ts` — typed operation documents (auto-generated)
- `services/api-gateway/src/lib/context.ts` — `GraphQLContext` interface (injected into all resolvers)

### Application Code
- `lib/api-spec/openapi.yaml` — existing Orval OpenAPI spec (Orval-only; kept for backward compat)
- `lib/db/src/schema/` — Drizzle table definitions (compatibility layer; Prisma is canonical going forward)
- `artifacts/api-server/src/routes/` — Express route handlers (dashboard, workflows, executions, connectors, apps)
- `artifacts/flowcraft/src/` — React frontend
- `lib/api-client-react/src/generated/` — generated React Query hooks
- `lib/api-zod/src/generated/` — generated Zod schemas for server validation

## Architecture decisions

- **OpenAPI-first at gateway boundary**: spec at `services/api-gateway/openapi/openapi.yaml` is the source of truth for all REST types; never hand-write `gateway-api.d.ts`
- **Prisma is the canonical ORM** (Section 29.2): `packages/shared-db/prisma/schema.prisma` covers all 40+ tables; Drizzle at `lib/db/` is a compatibility layer and should not be extended
- **Drizzle compatibility layer**: keep existing Drizzle schema files as-is; do not add new tables to Drizzle
- **GraphQL SDL is the contract**: `services/api-gateway/graphql/schema.graphql` is authoritative; `codegen.ts` produces 3 outputs: base types, resolver types, operation types
- `connectors/categories` route placed BEFORE `connectors/:id` in router to avoid Express matching "categories" as an id
- `apps/stats` route placed BEFORE `apps/:id` for same reason
- Workflow run endpoint simulates execution with random success/failure and inserts a real execution row
- App stats endpoint computed on-the-fly from the apps table (no separate stats table)
- Root `pnpm run codegen` now covers all three codegen pipelines in sequence

## Product

- **Workflow automation**: Create, manage, and run multi-step workflows triggered by schedules, webhooks, or third-party events
- **Connector Marketplace**: 15 pre-loaded connectors (Slack, GitHub, Stripe, Google Sheets, etc.) with install flow
- **Internal Tools Builder**: Create and manage dashboards, CRUD apps, forms, and reports
- **Execution History**: Full audit log of all workflow runs with step-level detail

## Gotchas

- Route order matters: `/connectors/categories` must be registered before `/connectors/:id`
- Route order matters: `/apps/stats` must be registered before `/apps/:id`
- After any OpenAPI spec change, run `pnpm run codegen` (covers all three pipelines)
- `pnpm add` from a subdirectory triggers corepack SIGABRT; always add deps to `package.json` directly then run `pnpm install` from workspace root
- Prisma `Unsupported("vector(1536)")` is required for pgvector columns (Prisma has no native pgvector type yet)
